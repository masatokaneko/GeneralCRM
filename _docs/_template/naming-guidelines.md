# 📝 命名ガイドライン

このドキュメントは、実装ドキュメントのファイル名とタスク名の命名規則を定義します。

## 📋 基本原則

### 1. 簡潔性

- 短く、わかりやすい名前を使用
- 不要な修飾語は避ける
- 20-30 文字程度を目安とする

### 2. 一意性

- 他のタスクと区別できる固有の名前
- 同じ名前の重複を避ける
- プロジェクト内で一意であること

### 3. 説明性

- 内容が推測できる名前
- 技術要素や目的を含む
- 実装範囲が明確になる名前

### 4. 一貫性

- 同じ種類のタスクは同じ命名パターンを使用
- プロジェクト全体で統一されたルール
- チーム内での共通理解

## 🎯 テンプレート別命名ルール

### 1. 一般実装 (`[タスク名] 実装`)

#### 命名パターン

**機能実装**

- 形式: `[機能名]機能実装`
- 例: `ユーザー認証機能実装`、`ファイルアップロード機能実装`
- 用途: 新しい機能の追加実装

**システム構築**

- 形式: `[システム名]システム構築`
- 例: `CI/CDパイプライン構築`、`監視システム構築`
- 用途: インフラやシステム全体の構築

**設定・環境**

- 形式: `[対象]環境設定`
- 例: `開発環境設定`、`本番環境設定`
- 用途: 環境構築や設定変更

**ライブラリ導入**

- 形式: `[ライブラリ名]導入`
- 例: `Storybook導入`、`ESLint導入`
- 用途: 新しいライブラリやツールの導入

**API 実装**

- 形式: `[API名]API実装`
- 例: `ユーザー管理API実装`、`ファイル操作API実装`
- 用途: バックエンド API の実装

**データベース関連**

- 形式: `[対象]DB[操作]`
- 例: `ユーザーテーブル設計`、`マイグレーション実装`
- 用途: データベース設計や操作の実装

### 2. コンポーネント実装 (`[コンポーネント名] コンポーネント実装`)

#### 命名パターン

**基本ルール**

- **PascalCase**: コンポーネント名は PascalCase で記述
- **機能説明**: コンポーネントの主要機能を含む
- **Atomic Design 分類**: 必要に応じて分類を含む

**単体コンポーネント**

- 形式: `[コンポーネント名] コンポーネント実装`
- 例: `Button コンポーネント実装`、`Modal コンポーネント実装`
- 用途: 単一のコンポーネント実装

**複合コンポーネント**

- 形式: `[コンポーネント名]群 コンポーネント実装`
- 例: `フォームコンポーネント群実装`、`ナビゲーションコンポーネント群実装`
- 用途: 関連する複数のコンポーネント実装

**ページコンポーネント**

- 形式: `[ページ名]Page コンポーネント実装`
- 例: `DashboardPage コンポーネント実装`、`UserProfilePage コンポーネント実装`
- 用途: ページレベルのコンポーネント実装

**レイアウトコンポーネント**

- 形式: `[レイアウト名]Layout コンポーネント実装`
- 例: `DashboardLayout コンポーネント実装`、`AuthLayout コンポーネント実装`
- 用途: レイアウト用のコンポーネント実装

### 3. リファクタリング (`[リファクタリング対象] リファクタリング`)

#### 命名パターン

**コンポーネントリファクタリング**

- 形式: `[コンポーネント名] リファクタリング`
- 例: `ダッシュボードコンポーネント リファクタリング`、`フォームコンポーネント群 リファクタリング`
- 用途: 既存コンポーネントの構造改善

**システムリファクタリング**

- 形式: `[システム名] リファクタリング`
- 例: `認証システム リファクタリング`、`API層 リファクタリング`
- 用途: システム全体の設計改善

**アーキテクチャリファクタリング**

- 形式: `[アーキテクチャ名] リファクタリング`
- 例: `Atomic Design準拠 リファクタリング`、`状態管理アーキテクチャ リファクタリング`
- 用途: 設計原則やアーキテクチャの適用

**パフォーマンスリファクタリング**

- 形式: `[対象]パフォーマンス改善 リファクタリング`
- 例: `画像読み込みパフォーマンス改善 リファクタリング`
- 用途: パフォーマンス向上を目的とした改善

## ⚠️ 命名時の注意点

### 避けるべき命名

#### 曖昧な表現

- ❌ `新機能実装`
- ❌ `バグ修正`
- ❌ `改善`
- ❌ `コンポーネント作成`
- ❌ `システム更新`

#### 日付のみ

- ❌ `2025年1月実装`
- ❌ `今月のタスク`
- ❌ `第1四半期対応`

#### 番号のみ

- ❌ `タスク1`
- ❌ `実装その2`
- ❌ `フェーズ3`

#### 長すぎる名前

- ❌ `ユーザー管理システムの認証機能とプロフィール編集機能を含む包括的な実装`
- ❌ `ダッシュボードページのサイドバーナビゲーションコンポーネントのレスポンシブ対応リファクタリング`

### 推奨する命名

#### 具体的

- ✅ `ユーザープロフィール編集機能実装`
- ✅ `ファイルアップロード機能実装`
- ✅ `OAuth認証システム構築`

#### 目的明確

- ✅ `パフォーマンス改善リファクタリング`
- ✅ `アクセシビリティ対応実装`
- ✅ `セキュリティ強化実装`

#### 技術要素

- ✅ `React Hook Form導入`
- ✅ `Tailwind CSS v4移行`
- ✅ `TypeScript厳密化設定`

#### 範囲明確

- ✅ `認証コンポーネント群実装`
- ✅ `ダッシュボード機能実装`
- ✅ `管理画面UI実装`

## 🔄 バージョン管理

同じ対象に対して複数回の作業を行う場合の命名パターン：

### 段階的実装

```
# 初回実装
ユーザー認証機能実装.md

# 機能追加
ユーザー認証機能拡張.md

# 改善・最適化
ユーザー認証機能改善.md

# 大幅な変更
ユーザー認証機能リファクタリング.md
```

### バージョン番号付き

```
# 大きな変更がある場合
ユーザー認証機能実装v2.md
API設計v3.md
```

### 特定用途

```
# 特定の目的がある場合
ユーザー認証機能セキュリティ強化.md
ユーザー認証機能パフォーマンス改善.md
ユーザー認証機能アクセシビリティ対応.md
```

## 🎯 命名チェックリスト

新しいタスクの命名時に以下をチェックしてください：

### 基本チェック

- [ ] 20-30 文字程度の適切な長さか？
- [ ] 内容が推測できる具体的な名前か？
- [ ] プロジェクト内で一意の名前か？
- [ ] 適切なテンプレート分類に従っているか？

### 内容チェック

- [ ] 実装対象が明確に示されているか？
- [ ] 技術要素が含まれているか（必要に応じて）？
- [ ] 目的や効果が伝わるか？
- [ ] 範囲が適切に限定されているか？

### 形式チェック

- [ ] 適切な命名パターンに従っているか？
- [ ] PascalCase/camelCase が正しく使われているか？
- [ ] 日本語として自然な表現か？
- [ ] ファイル名として使用可能な文字のみか？
